# ADR 0003: Durable Server-Side Full-Catalog Screening

- Status: Accepted for v2.1 implementation
- Date: 2026-07-20

## Context

Version 2.0 screens one selected object against the browser catalog in a module Web Worker. Full-catalog screening has materially greater CPU, memory, persistence, cancellation, recovery, and audit requirements and cannot be made reliable by expanding the browser loop.

The application already has a local Python standard-library server and a locked Node/satellite.js implementation of the tested TLE propagation and TCA path. Static deployment must continue to work without a server, while server-backed screening must be durable, bounded, reproducible, and authenticated for mutation.

## Decision

1. The Python server owns authenticated `/api/v1` admission, SQLite persistence, job state, retry/timeout policy, cancellation, pagination, and Server-Sent Event delivery.
2. A locked Node subprocess owns catalog validation, satellite.js propagation, conservative broad-phase candidate generation, and synchronized TCA refinement. It emits structured progress and atomically writes an immutable result artifact before the server imports candidates, events, errors, and metrics into SQLite.
3. The broad phase uses time-partitioned swept trajectory bounds in a three-dimensional spatial hash. Each object chord is expanded by the configured screening radius and acceleration-bound curvature allowance. Pairs are then checked with the relative-chord bound before refinement. Global unbounded all-pairs iteration is prohibited.
4. Catalog revisions are content-addressed and copied into a private runtime directory. Lifecycle observations preserve new, updated, absent, reappeared, decayed, and retired states without merging objects by similar names.
5. Job requests are idempotent by an explicit key and normalized request hash. A reused key with a different request is rejected.
6. Mutating and job-detail routes require a configured bearer token. The server remains loopback-only by default; public deployment still requires a reviewed TLS proxy, stronger identity/authorization, quotas, monitoring, and operational ownership.
7. Existing v2.0 conjunction event payloads remain readable. Version 2.1 adds separate job, catalog-revision, candidate, pagination, and API contracts rather than silently changing stored v2.0 event meaning.
8. Static hosting keeps the v2.0 selected-object Worker and clearly reports that durable full-catalog jobs require the optional authenticated server.

## Consequences

- SQLite WAL provides a reproducible single-node development and local-analyst store without adding a mandatory Python package dependency. It is not a distributed queue.
- Node subprocess isolation enables cancellation and timeout enforcement and reuses the locked scientific path, but process startup and catalog transfer add overhead.
- Restart recovery requeues interrupted attempts deterministically from the immutable input snapshot. It does not claim distributed exactly-once execution.
- The spatial hash must be validated against brute force on tractable deterministic catalogs. Exceeding cell-density, pair-check, candidate, memory, or time limits fails or marks the job partial; it must never be reported as complete coverage.
- OMM/OEM adapters preserve their original messages, frames, time systems, and precision. Only formats supported by an explicit propagation adapter are eligible for screening.
- This architecture does not add collision probability, operational alerts, or maneuver advice. All v2.1 output remains Experimental and non-operational.
